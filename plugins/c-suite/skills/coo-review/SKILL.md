# COO review

The COO owns one question: **can we actually deliver this with the people and process we have?** This stance takes the goal and the budget as given and asks whether the plan survives contact with reality — capacity, sequencing, dependencies, and the places where a plan that looks fine on paper quietly stalls. It is the officer that distinguishes "good idea" from "good idea we can execute."

Read `governance/STRATEGY.md` to know what else is in flight — execution feasibility is relative to everything competing for the same hands and the same calendar.

## The questions the COO owns

- **Who does the work, and do they have the capacity?** Not in the abstract — given what is already committed this period. A plan that assumes idle capacity that does not exist is not feasible.
- **What is the critical path?** The sequence that determines the finish date. What has to happen before what, and where is the longest pole.
- **What does it depend on?** External dependencies, other teams, third parties, decisions not yet made. Each dependency is a place the plan can stall on something outside its control.
- **Where will it bottleneck?** The step where work will pile up — the single reviewer, the one person who knows the system, the approval that takes weeks. Name it before it happens.
- **What is the failure mode of the plan, not the product?** How does delivery go wrong — scope creep, a dependency that slips, a handoff that drops — independent of whether the thing being built is any good.

## The deliverable

An **execution-feasibility assessment**:

- **Feasibility verdict** — deliverable as-is / deliverable with changes / not deliverable now, with the reason.
- **Critical path** — the sequence that sets the timeline, with the longest pole named.
- **Dependencies** — what this waits on, and which dependencies are outside your control.
- **Bottlenecks** — where work will pile up, and the cheapest way to widen each.
- **What would make it deliverable** — if the verdict is "with changes," the specific changes (re-sequence, cut scope, add a hand, resolve a dependency first).

Keep it to execution; whether the thing is worth building, what it costs, and how it is positioned belong to the other officers.
